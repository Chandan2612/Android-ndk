01 / Meaning

What does NDK mean?

NDK means Native Development Kit. It is a toolkit from Android that lets you write part of your app using native languages like C and C++ instead of only Java or Kotlin.

📱

Normal Android layer

Most Android apps are written in Kotlin or Java. This layer is easy for UI, API calls, database work, permissions, and app screens.

⚙️

Native layer

Native code means low-level C/C++ code that runs closer to the device CPU. Android packages this code as .so files.

🌉

JNI bridge

Java/Kotlin cannot directly call C/C++ normally. JNI works like a bridge between Android code and native code.

02 / Analogy

Simple real-life example

Think of your Android app like a restaurant. Kotlin/Java is the waiter and UI. C/C++ native code is the kitchen machine doing heavy work very fast. JNI is the door between the waiter and the kitchen.

User taps button App receives action from UI
Kotlin/Java Handles Android app logic
JNI Connects managed code to native code
C/C++ Runs fast native calculation
Result returns UI shows final output
03 / Why

Why do we use NDK?

You do not use NDK for every app. You use it only when native C/C++ gives a real benefit.

🚀

Good use cases

  • Game engines and graphics-heavy apps
  • Video processing, audio processing, image processing
  • Machine learning native libraries
  • Reusing existing C/C++ libraries
  • Encryption, compression, and performance-heavy algorithms
  • Flutter/React Native plugins that include native SDKs
⚠️

Bad use cases

  • Simple UI screens
  • Normal API calls
  • Firebase, login, CRUD, forms, dashboards
  • Basic app business logic
  • Using NDK only because it sounds advanced
04 / Build Flow

How NDK code becomes part of an APK/AAB

The important idea: C/C++ source code is compiled into native shared libraries. Android stores these files as .so files inside ABI-specific folders.

C/C++ file native-lib.cpp
NDK compiler Compiles native code
.so library libnative-lib.so
ABI folder arm64-v8a, x86_64
APK/AAB Final Android build
05 / Terms

Important NDK terms you must know

These words appear again and again when you debug Android native library problems.

📦

.so file

A .so file means shared object library. It is the compiled native C/C++ code used by Android.

🧬

ABI

ABI means Application Binary Interface. Simple meaning: CPU type compatibility. Example: arm64-v8a for most modern Android phones.

🛠️

CMake

CMake tells Gradle how to build C/C++ files into .so native libraries.

🌉

JNI

JNI means Java Native Interface. It lets Kotlin/Java call C/C++ functions and receive results back.

📁

jniLibs

A folder where prebuilt native libraries can be placed, usually like android/app/src/main/jniLibs/arm64-v8a/.

📐

16 KB page size

Newer Android devices may use 16 KB memory pages. Native libraries must be built/packaged correctly to support it.

06 / Flutter

Where NDK appears in Flutter apps

In Flutter, you may not write C/C++ yourself, but your app can still contain native libraries because plugins may include them.

🧩

Plugin native libraries

Packages like video players, webviews, ML, camera, encryption, and some ad/analytics SDKs can include native .so files. Your Flutter app then ships those libraries inside the Android build.

🔥

Common issue

Sometimes your app fails Play Console checks or crashes on certain devices because one native library is outdated, built for wrong ABI, or not compatible with required memory page size.

07 / Gradle

Important Gradle configuration

This is the kind of Android config you often see in Flutter projects when native libraries are involved. 

android {
    namespace "com.example.my_app"
    compileSdk 35

    defaultConfig {
        applicationId "com.example.my_app"
        minSdk 23
        targetSdk 35

        // This tells Android which NDK version to use for native builds.
        ndkVersion "28.0.12433566"

        // ABI filters decide which CPU architectures your app supports.
        ndk {
            abiFilters "arm64-v8a", "x86_64"
        }
    }

    packagingOptions {
        jniLibs {
            // false means native libraries can stay uncompressed.
            // This is usually better for modern Android native library loading.
            useLegacyPackaging false
        }
    }
}
08 / JNI Example

Small example: Kotlin calls C++

This is only to understand the idea. Kotlin declares a native function. C++ implements it. JNI connects them. 

// MainActivity.kt
class MainActivity : AppCompatActivity() {

    // Load the compiled native library.
    companion object {
        init {
            System.loadLibrary("native-lib")
        }
    }

    // This function is implemented in C++.
    external fun addNumbers(a: Int, b: Int): Int

    fun calculate() {
        val result = addNumbers(10, 20)
        println(result) // Output: 30
    }
}
// native-lib.cpp
#include <jni.h>

// JNI function name format:
// Java_packageName_ClassName_functionName
extern "C"
JNIEXPORT jint JNICALL
Java_com_example_myapp_MainActivity_addNumbers(
        JNIEnv* env,
        jobject object,
        jint a,
        jint b
) {
    // Return result back to Kotlin/Java.
    return a + b;
}
09 / Packaging

What is useLegacyPackaging?

This setting controls how native .so libraries are packaged inside your APK. It becomes important when Android loads native libraries.

✅

useLegacyPackaging false

Native libraries can remain uncompressed and page-aligned inside the APK. This is commonly preferred for modern Android builds, especially when dealing with newer native library requirements.

🧯

useLegacyPackaging true

Native libraries are extracted in the older style. It can help in some old compatibility cases, but it is not always the best choice for modern Play Console native library checks.

10 / Debugging

Common NDK problems and simple meaning

When native issues happen, the error message looks scary. Usually it is about missing files, wrong ABI, old libraries, or Gradle config.

Simple meaning: Android tried to load a native .so library, but it could not find it or it could not load it.

  • Check if the .so file exists inside the correct ABI folder.
  • Check System.loadLibrary("library-name").
  • Check whether plugin supports your target device ABI.

Simple meaning: your app supports arm64-v8a, but one native library does not provide an arm64-v8a version.

  • Update the plugin or SDK that provides the native library.
  • Avoid 32-bit-only native libraries in modern apps.

Simple meaning: one or more native libraries may not be compatible with Android devices using 16 KB memory pages.

  • Use newer NDK/toolchain versions.
  • Update third-party plugins that include .so files.
  • Prefer modern packaging and 64-bit libraries.

Simple meaning: shipping native libraries for many ABIs increases APK size.

  • Use App Bundle so Play Store delivers only required ABI to users.
  • Use ABI filters carefully for testing builds.
11 / Roadmap

NDK zero-to-hero roadmap

Follow this order. Do not jump directly into advanced native debugging before understanding the basics.

Level 1: Understand why NDK exists

Learn the difference between Android Kotlin/Java code and native C/C++ code. Understand when NDK is useful and when it is unnecessary.

Level 2: Learn ABI and .so files

Understand arm64-v8a, armeabi-v7a, x86, x86_64, and why every CPU architecture needs its own native binary.

Level 3: Learn Gradle native config

Study ndkVersion, abiFilters, packagingOptions, jniLibs, externalNativeBuild, and CMakeLists.txt.

Level 4: Build one tiny JNI demo

Create one Kotlin function that calls one C++ function. This makes the complete bridge clear.

Level 5: Debug Flutter native library issues

Inspect APK/AAB contents, check plugin .so files, update old SDKs, and solve ABI/page-size warnings.

Level 6: Advanced native performance

Learn memory management, threading, profiling, CMake optimization, and safe communication between managed and native code.

12 / Interview Ready

How to explain NDK in an interview

Use this simple answer when someone asks you what NDK is.

🎯

Best answer

Android NDK stands for Native Development Kit. It allows Android apps to use C and C++ code. We use it when we need high performance, existing native libraries, or low-level system work. The native code is compiled into .so files for different ABIs like arm64-v8a. Kotlin or Java communicates with native code through JNI. In Flutter apps, NDK problems often come from plugins that include native .so libraries.

14 / Final Summary

NDK in one simple picture

Your app does not become better just because it uses NDK. NDK is useful only when native code solves a real problem.

🧠

Remember this

Kotlin/Java is best for normal Android app work. C/C++ is for performance-heavy or reusable native code.

📦

Native output

C/C++ becomes .so files. These files must match the device CPU architecture.

🔍

Flutter reality
